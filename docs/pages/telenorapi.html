Authentication

\n

Parts of the API requires authentication, in order to reliably identify the\n user to a specific customer record. In this initial version, the API uses HTTP\n BASIC authentication with username and password. The API offers a number of\n methods to assist in building the username and password registration flow as\n well as convenience methods to help recover lost passwords.\n

\n

As an option, API access may also be guarded with a subscription, requiring\n the authenticated customer to hold an active subscription on multi-screen\n devices to gain access.\n

\n

Unauthenticated users can use other parts of the API anonymously, e.g. to\n retrieve EPG programming data and other, non-personalized information.\n

\n

Some services in the API are “personal” and hence require authentication, while others are\n consideret public, and may be accessed anonymously. A few services may be accessed both ways,\n and support extra features in the authenticated version. E.g., the channels service can be\n queried anonymously, to retrieve the selection of available channels, but if it is called\n with authentication and an extra query parameter, it can also be used to determine exactly which\n channels can be viewed with respect to the customer's current subscriptions profile.

\n

All requests that require authentication use the URI containing …/auth/rs/….\n Anonymous requests use …/rs/….\n

\n
\n
…/client-portal/rs
\n
For anonymous calls. Responses may be cached in a shared manner, e.g. by\n a proxy cache, returning the same cached content to different clients. Authentication\n headers, if present, are ignored.
\n
…/client-portal/auth/rs
\n
For authenticated calls. HTTP Basic Authentication headers are validated\n against the FokusOn customer login tables. Cached responses should not be shared\n by different clients.

Error handling

\n

Many services have one or more error modes, in addition to the 'normal' mode. Error modes\n include cases of regular use such as data validation failures, insufficient access privileges, etc.\n as well as system level cases such as multi-user interference, system resource exhaustion, time-outs,\n etc.\n All cases are reported using standard\n HTTP status codes, possibly with extra error information returned in custom HTTP headers and/or\n the response body itself.

\n
\n
Normal mode
\n

In regular, successful use, all services return one of the\n 2xx Success or\n 3xx Redirection codes.\n GET methods will always return either 200 OK, or 304 Not Modified if called with an 'if modified since'\n condition.

\n

POST and PUT methods may also return 201 Created together with a Location header for the address of\n the newly created entity.

\n

DELETE methods may also return 204 No Content.

\n
\n
User/Application level errors
\n

Error conditions reported from the application business logic will always return one of the\n 4xx Error codes,\n either as a self-descriptive code, where the cause of the failure is evident, e.g. 401 Unauthorized, 404\n Not Found, etc., or as the more generic 400 Bad Request, accompanied by an error message identifier and\n zero or more error message parameters, returned in 'X-Message-Key/-Param' custom HTTP response headers.

\n
\n
System level errors
\n
Error conditions outside of the application business logic, reported from the application server itself,\n or from the intermediary network and load balancing equipment, will usually be reported by one of the\n 5xx Server Error codes,\n or by some exceptional client-side condition, depending on the API client library being used, the details of\n which is beyond the scope of this document.\n

Pagination

\n

Pagination is supported on some of the service methods. Currently, there are two types of pagination\n metaphors implemented: random access and sequential. The random-access was implemented as first, but due its\n limitations the sequential-access mechanism was implemented later. Any pageable service methods that use\n the Random Access pagination can be expected to be changed to use the Sequential pagination in the future.\n

\n
Random-Access Pagination
\n

There are two additional query parameters\n that any such method recognizes: \"start\" and \"limit\". The \"start\" parameter allows the client to skip\n over the given number of items, meanwhile the \"limit\" parameter determines the maximum number of items\n to be returned. The default values are 0 for \"start\" and 'all' for \"limit\".\n

\n

\n Invalid numeric values for these query parameters will be ignored and the method will use the default values.\n Non-numeric values will cause an HTTP 404 error.\n

\n
Sequential Pagination
\n

There are two additional, optional query parameters\n that any such method recognizes: \"pgSize\" and \"pg\". The \"pgSize\" parameter determines the page size - i.e.,\n the maximum number of items to be returned. The \"pg\" parameter conveys the paging context, and generally it is not\n expected to be generated by the client. Instead, the client is should follow the HATEOAS principle\n and use pre-generated URIs which already contain this parameter. These URIs can be obtained from the response headers;\n here is an example of a set of URIs obtained from the response:\n

    \n
  • Link-Next: /client-portal/rs/vod/shops/11/prices?pg=f95511&pgSize=10&currency=DKK
    • \n
  • Link-Prev: /client-portal/rs/vod/shops/11/prices?pg=b95471&pgSize=10&currency=DKK
    • \n
  • Link-First: /client-portal/rs/vod/shops/11/prices?pg=f&pgSize=10&currency=DKK
    • \n
  • Link-Last: /client-portal/rs/vod/shops/11/prices?pg=b&pgSize=10&currency=DKK
    • \n
\n

\n

\n Depending on the user action (Link-Next, Link-Prev, etc.), the client would then just have to pick and use the corresponding\n link for the next request - without having to construct the URI.\n

\n

\n If the \"pgSize\" parameter is not specified, the paging mechanism is not engaged and all results are returned.\n
